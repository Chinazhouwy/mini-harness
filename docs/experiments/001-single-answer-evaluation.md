# 实验 001：单题回答评测

## 背景

我想先验证：一个普通模型能否识别一段"基本正确但不完整"的 Java 面试回答，而不是直接给出泛泛的正面评价。

## 本次只验证什么

- 模型能否识别回答中的正确部分。
- 模型能否发现明显遗漏。
- 模型是否会误判或补充候选人没有说过的内容。
- 模型生成的追问是否对应真实缺口。

## 输入问题

线程池提交一个任务后的完整执行流程是什么？

## 候选人回答

任务提交后，线程池会先创建核心线程执行任务。
核心线程满了以后，任务会进入阻塞队列。
队列满了以后继续创建线程，达到最大线程数以后执行拒绝策略。

# 1. 配置环境变量
export AI_API_KEY="your-api-key"

## 模型原始输出

详见 [`target/experiments/001-single-answer-evaluation-output.md`](../../target/experiments/001-single-answer-evaluation-output.md)

**摘要**：模型将回答判定为"骨架正确，深度不足"，总体评分 6/10。

## 我的观察

### 判断正确的地方

- **准确识别了回答的骨架结构**：模型正确指出候选人描述了"核心线程 → 队列 → 最大线程 → 拒绝策略"的顺序，没有因为回答流畅就默认其完整。
- **发现了真正的知识缺口**：模型追问了具体的边界条件（如 `workerCount < corePoolSize`、不同队列类型的影响），这些问题确实能区分"会用线程池"和"理解线程池实现"。
- **指出了表述不够严谨的地方**：如"核心线程满了以后"应为"workerCount == corePoolSize"，"继续创建线程"应为"创建非核心线程"。

### 判断不准确的地方

- **评分可能偏慷慨**：6/10 对于一个只描述了骨架、没有任何实现细节的回答来说，可能偏高。真实的高级工程师面试中，这类回答通常不会及格。
- **部分遗漏其实是对中级和高级的不同期望**：模型列出了很多遗漏点（如 `execute()` 内部实现、不同队列类型），但有些内容对于"理解线程池"比"能正确使用线程池"要求更高。需要区分哪些是必须知道的、哪些是加分项。

### 追问是否有价值

追问质量很高。特别是：
- "如果阻塞队列是无界的，maximumPoolSize 还会起作用吗？" —— 这是一个能揭示是否真正理解参数之间关系的经典追问。
- "corePoolSize=2, max=5, 队列容量=3，提交第 3、4、6 个任务分别发生什么？" —— 通过具体数字验证理解。

### 模型是否有虚构

模型没有虚构候选人没有说过的内容。指出的遗漏点确实在候选人回答中不存在。模型严格区分了"候选人说了什么"和"候选人没说什么"。

## 暴露出来的问题

1. **评分缺乏锚定标准**：6/10 的判断没有参照物，不确定模型是如何得出这个分数的。需要有标准样本或 Rubric 来校准。
2. **遗漏点的优先级不明确**：有些遗漏（如 `keepAliveTime`）是重要知识缺口，有些（如 `ThreadFactory` 的具体作用）是加分项。当前评测没有区分"必须知道"和"最好知道"。
3. **Prompt 中"不要输出 JSON"的约束有效**，说明后续结构化输出是可行的方向。
4. **单次评测的稳定性未知**：同一段回答多次评测是否得到一致结论，当前没有验证。
