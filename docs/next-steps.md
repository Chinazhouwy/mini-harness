# MiniHarness 后续规划

> 这不是一份“把所有 AI 技术都做一遍”的路线图。
>
> 排序依据只有一个：先处理已经能从当前代码中看到、会妨碍继续学习或真实使用的问题；
> 后续功能必须由实际运行中出现的新问题推动。

## 当前判断

项目现在有两个学习入口：Spring AI API Demo，以及一个把意图识别、出题、评测、追问串起来的控制台原型。下一步最重要的不是继续堆功能，而是让这个原型能够稳定地跑完一轮、重启后不迷失上下文，并能让我们判断模型评测是否靠谱。

优先级含义：

| 优先级 | 含义 |
|---|---|
| P0 | 当前阻塞编译、运行或会造成明显错误行为的问题 |
| P1 | 原型能运行后，影响一次完整练习可信度的问题 |
| P2 | 已有主流程稳定后，减少混乱、便于验证和回看的优化 |
| P3 | 只有出现真实需要时才考虑的扩展，不提前实现 |

## P0：先让当前原型能编译、能重新开始

### P0-1：处理缺失的 `QuestionStatus`

**现象**：`InterviewQuestion` 引用了不存在的 `QuestionStatus`，当前 `mvn test` 在编译主代码时失败。

**涉及文件**：

- `src/main/java/com/chinazhouwy/miniharness/InterviewQuestion.java`

**需要先做的决定**：`InterviewQuestion` 是不是现在就要接入主流程？

- 若暂时不用它：移除或隔离这组未接入的领域草图，让 Demo 先恢复可运行。
- 若准备接入：再定义最小的题目状态，并让它服务于真实流程，而不是只为消除编译错误。

**完成标准**：`mvn test` 至少能完成编译阶段；没有真实 Key 时 LLM Demo 应跳过或不运行，而不是因为代码错误失败。

**暂不做**：不因为这个问题直接引入数据库、完整状态机或复杂领域模型。

### P0-2：迁移或清理旧版 `history.json`

**现象**：当前恢复逻辑期望 `StoredMessage(role, content)`，但仓库中的 `data/history.json` 仍是旧版 Spring AI `Message` 的序列化格式。

**涉及文件**：

- `data/history.json`
- `MiniHarnessDemo.loadHistoryInfo()`
- `MiniHarnessDemo.toStoredMessages()`

**建议**：学习阶段最简单的办法是先备份旧文件并删除，让程序按新格式生成；如果旧记录确实有保留价值，再单独写一次性迁移脚本。

**完成标准**：启动、输入一轮、退出后，`history.json` 能被下次启动正确读回，不再打印反序列化异常。

**暂不做**：不为两个本地 JSON 文件设计通用数据迁移框架。

### P0-3：给控制台入口补一个最小冒烟路径

**问题**：现在只有人工运行才能知道“模型配置错误、历史格式错误、意图结构化输出失败”发生在哪一步。

**建议**：先不 Mock LLM，只补一个不调用模型的启动检查，至少验证：Key 是否为空、`data/` 是否可写、历史 JSON 是否可读取。

**完成标准**：配置缺失或历史损坏时，控制台能给出明确中文提示，而不是把底层异常直接甩给用户。

## P1：让一次面试练习逻辑上完整

### P1-1：保存真正的会话状态，而不只保存消息历史

**现象**：重启后虽然能读回 `history`，但 `currentQuestion` 重新变为空、`InterviewState` 重新变为 `IDLE`。用户继续回答时，程序不再进入评测分支。

**涉及文件**：

- `MiniHarnessDemo.test()`
- `InterviewSession`
- `InterviewState`

**建议**：先用一个很小的本地 `session.json` 保存：当前题、当前阶段、历史和 attempts。此时正好验证 `InterviewSession` 是否真的需要从草图变成可用对象。

**完成标准**：在“已出题、等待回答”时退出并重启，下一次输入回答仍会被评测，并且追问围绕同一道题。

**暂不做**：不引入 PostgreSQL；先证明“会话需要恢复”这个问题真实存在。

### P1-2：让状态真正约束流程

**现象**：`InterviewState` 当前只打印日志，不阻止非法操作。例如没有出题时输入一段技术回答，流程仍会进入 `ANSWER` 分支；`FINISHED` 也未被实际设置。

**建议**：在 `switch` 前或每个分支中写最小规则：

```text
IDLE             -> 允许请求下一题、普通聊天、退出
WAITING_ANSWER   -> 允许回答、提示、讲解、退出
DISCUSSING       -> 允许追问、下一题、讲解、退出
FINISHED         -> 不再调用模型
```

**完成标准**：每个不允许的输入都有明确提示，而且不会额外调用模型。

**暂不做**：不引入 Spring StateMachine；先验证几个 Java `if`/`switch` 是否已经足够。

### P1-3：对齐意图枚举、Prompt 和分支

**现象**：代码里有 `FOLLOW_UP` 和 `CHAT`，但意图识别 Prompt 没有要求模型返回它们，`switch` 也没有实现对应分支。

**建议**：每增加一个意图，必须同时完成三件事：Prompt 声明、结构化结果测试、Java 分支处理。当前如果还不需要这两个能力，就先删掉预留值，或明确返回“暂未支持”。

**完成标准**：枚举中的每个值都有清楚的产生条件和处理结果；没有“模型能返回但程序只会 default”的悬空分支。

### P1-4：让评测输入明确且可校验

**现象**：评测器通过整段 `history` 推断当前题和当前回答，能工作但边界模糊；`AnswerEvaluation` 也没有校验分数范围、空字符串和异常输出。

**建议**：先把“当前题”和“当前回答”显式拼进评测 Prompt，并增加最小校验：分数应在 0 到 10、`missingPoint` 允许空、结构化转换失败时输出原始响应或可读错误。

**完成标准**：给定同一题和同一回答时，日志中能明确看到评测依据；模型返回明显不合法数据时不会悄悄写进 `attempts.json`。

**暂不做**：不急着设计完整 Rubric、多个维度或复杂评分算法。

### P1-5：修正保存时机（已完成）

**现象**：数据在下一次读输入前保存；回答后立刻输入 `exit` 时，最后一轮可能没有落盘。

**已实施**：普通业务分支在本轮处理完成后立即保存；本地 `exit` 和模型识别出的 `EXIT` 都会在退出前显式保存。

**已验证范围**：代码路径上，出题、回答、评测、追问、聊天和退出都不会再依赖“下一次输入”才触发保存。实际恢复效果仍应在下一次控制台实验中验证。

## P2：让“模型判断”开始可比较

### P2-1：为一题建立人工 Rubric

先只选一道足够熟悉的题，例如线程池任务提交流程，手工写清：

- 必须覆盖的关键点；
- 常见误解；
- 哪些缺口值得追问；
- 哪些表达可以接受。

**完成标准**：评测 Prompt 不再只说“请打分”，而是拿着明确的参考标准判断。

### P2-2：把评测结果从“一个分数”扩成“结论 + 证据”

建议先新增而不是直接推翻 `AnswerEvaluation`：

```text
coveredPoints      回答中已经提到的点
missingPoints      没有提到的关键点
incorrectClaims    明确错误的说法
answerQuote        候选人原话
explanation        为什么这句原话构成证据或错误
```

**完成标准**：看到一个遗漏或错误时，能回到具体回答内容，而不是只看到模型结论。

### P2-3：最小 Eval 集

围绕同一道题准备 6 到 10 个固定回答：完整、部分正确、明显错误、术语很多但缺关键点、表达差但核心正确、自相矛盾。

每次改 Prompt、模型或结构化字段后重跑它们，人工记录：哪些结果变好、哪些退化。

**完成标准**：能回答“这次改 Prompt 后为什么认为它更好”，而不只凭一次主观试玩。

### P2-4：控制模型上下文大小

**问题**：当前每次调用都把完整 `history` 发给模型，练习久了会增加 token、延迟和干扰。

**建议**：先记录消息数量和 token usage，再决定是否只保留最近 N 轮；不要立刻做复杂自动摘要。

**完成标准**：能观察到上下文增长，而不是盲目猜测；若实施截断，不影响当前题和最近一次评测的理解。

### P2-5：补不依赖真实模型的单元测试

优先测试确定性的 Java 行为：状态迁移、保存/加载转换、非法意图、评测结果校验。真实模型调用保留为 `@Tag("llm")` 的实验测试。

**完成标准**：普通 `mvn test` 不花 API 费用，也能验证大部分业务规则。

## P3：只在需求出现后再做的功能

### 会话与数据层

当本地 JSON 已经妨碍多次训练回放、并发或查询时，再评估 PostgreSQL、Flyway 和 `interview_session` / `interview_turn` / `evaluation_run` 等表。

### 面试训练能力

当单题评测和追问稳定后，再考虑：按薄弱点选题、长期能力证据、复盘报告、多主题题库。

### Mini Harness 抽象

只有当评测、追问、报告等多个场景真的重复遇到“模型调用生命周期、状态、事件、工具循环”问题时，再抽 `AgentState`、Hook、Event、ContextPolicy。不要先做一个空的通用 Runtime。

### 工具调用、RAG、MCP、多 Agent

这些能力都应由具体问题触发。例如题目资料需要动态查询时再接工具；题库真的大到检索困难时再考虑 RAG；不因为名字听起来高级而提前塞进来。

## 每次继续开发前的记录模板

每做完一个 P0/P1/P2 项，在 `docs/experiments/` 新建或补充一篇记录：

```markdown
# 实验 N：一句话说明本次只验证什么

## 观察到的问题

## 本次改动

## 如何验证

## 实际结果

## 还没解决什么

## 这个结果决定的下一步
```

这样这份规划不会变成死板的任务表：如果下一次真实运行暴露出的问题更重要，可以调整顺序；但 P0 的“先恢复编译和基本可运行”不应被跳过。
