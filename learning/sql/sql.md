简单来说，**窗口函数**是一种在SQL或数据分析中，能够在不改变数据行数的情况下，对每一行数据在其指定的一个“窗口”（即一组相关联的行）上进行计算的函数。

你可以把它想象成一个**带有滑动计算框的表格**。这个“窗口”会滑过每一行数据，并计算出一个基于窗口内数据的结果，然后附加到当前行上。

为了更好理解，我们把它和普通的**聚合函数**（如 `SUM`, `AVG`, `MAX`）做个对比：

| 特性 | 普通聚合函数 (GROUP BY) | 窗口函数 |
| :--- | :--- | :--- |
| **核心操作** | 将多行**折叠**成一行汇总结果。 | 为**每一行**计算，并**保留所有原始行**。 |
| **结果行数** | 通常少于原表行数。 | 与原表行数**完全相同**。 |
| **典型例子** | 计算每个部门的平均工资，输出一个汇总表（部门，平均工资）。 | 计算每个部门的平均工资，并将结果附加到每个员工的记录旁边。 |
| **应用场景** | 生成报表、统计摘要。 | 计算排名、累计和、环比、同比、移动平均等。 |

### 一个具体的例子

假设有一张“员工工资表”：

| 姓名 | 部门 | 月薪 |
| :--- | :--- | :--- |
| 张三 | 销售部 | 10000 |
| 李四 | 销售部 | 12000 |
| 王五 | 技术部 | 15000 |
| 赵六 | 技术部 | 18000 |

**需求**：查询所有员工信息，并同时显示他所在部门的平均工资。

-   **不用窗口函数**：逻辑比较复杂。需要先聚合查询部门平均工资，再通过关联查询把两个结果拼起来，SQL语句比较繁琐。
-   **使用窗口函数**：可以非常直观地写出如下SQL：

```sql
SELECT
    姓名,
    部门,
    月薪,
    AVG(月薪) OVER (PARTITION BY 部门) AS 部门平均工资
FROM
    员工工资表;
```

**结果**：

| 姓名 | 部门 | 月薪 | 部门平均工资 |
| :--- | :--- | :--- | :--- |
| 张三 | 销售部 | 10000 | **11000** |
| 李四 | 销售部 | 12000 | **11000** |
| 王五 | 技术部 | 15000 | **16500** |
| 赵六 | 技术部 | 18000 | **16500** |

你看，`AVG(月薪) OVER (PARTITION BY 部门)` 这个窗口函数，按“部门”划分了窗口，在每个窗口内计算平均工资，然后把结果原封不动地附在每一行后面，完美满足了“不变行数，附加计算”的需求。

### 窗口函数的三大核心组成部分

一个窗口函数通常写成 `函数(...) OVER (PARTITION BY ... ORDER BY ...)`，其中：

1.  **函数(Function)**：你想执行的计算，比如 `ROW_NUMBER()`（排名），`SUM()`（求和），`LAG()`（取上一行的值）等。
2.  **分区(PARTITION BY)**：将数据分成几个独立的“组”或“窗口”。如果不写，整个表就是一个窗口。
3.  **排序(ORDER BY)**：定义在每个窗口内，数据计算的顺序。对于排名、累计等函数至关重要。

### 总结

-   **核心思想**：**保留所有原始行**，同时进行跨行计算。
-   **最大优点**：极大地简化了SQL代码，尤其在处理排名、同比、环比、移动平均、累计和等问题时，比传统的自关联或子查询方式**更简洁、高效、易读**。
-   **一句话记住**：用 `GROUP BY` 会把多行压成一行，而用窗口函数 `OVER()` 可以在每一行上“放眼望去”，看到它所在的那个“窗口”里的信息。
